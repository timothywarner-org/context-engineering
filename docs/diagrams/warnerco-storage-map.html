2. Storage cheat sheet

Format File / location What lives there CoALA tier Read pattern
JSON
flat file		 data/schematics/schematics.json 25 robot schematics — the source of truth. Loaded into RAM at startup. Semantic Keyword match on text fields. Fastest to start, no embeddings.
Chroma
vector DB		 data/chroma/chroma.sqlite3 + segment dirs Same 25 schematics, embedded as dense vectors. Indexed once via ChromaMemoryStore.index_all(). Semantic Cosine similarity over embeddings — finds similar things.
Azure AI Search
cloud index		 External Azure resource (no local file) Production-grade replacement for Chroma. Same schematics, hybrid keyword + vector. Semantic Same role as Chroma, just enterprise-deployed.
SQLite
scratchpad		 data/scratchpad/notes.db Session-scoped observations and inferences (predicate, subject, object, content). Accumulates across the conversation. Working Filter by subject/age, inject top-N into prompt context.
SQLite
episodic		 data/episodic/events.db Timestamped events (turns, tool calls, observations) with importance scores. Episodic Park et al. recall: α·recency + α·importance + α·relevance.
SQLite
knowledge graph		 data/graph/knowledge.db 117 entities + 221 relationships as (subject, predicate, object) triplets. NetworkX in front of SQLite. Graph overlay Traversal — finds connected things (neighbors, shortest path).
Python source
decorators		 app/mcp_tools.py (@mcp.prompt()) Versioned skills/workflows ("how to compare two robots", "how to diagnose"). Code is the storage; git is the version control. Procedural Listed at memory://procedural-catalog. Invoked by the user, not the pipeline.

3. JSON vs Chroma vs Azure — the semantic-memory choice

All three are different implementations of the same CoALA tier (semantic memory). You pick one with MEMORY_BACKEND in .env or mcp.json. The tools and API shapes do not change.

4. SQLite is doing three different jobs

SQLite shows up three times because it is the right tool for each — local, embedded, durable, fast. But each .db file is a different memory tier with a different schema.

6. Consolidation — the "sleep cycle"

Working + episodic memory are short-lived. Periodically (or on demand via warn_consolidate_memory), the agent reads them, distills durable facts via ctx.sample(), and writes them back to the semantic store as synthetic FACT-* records.

  classDef episodic fill:#d2a8ff,stroke:#d2a8ff,color:#0e1117

  SCRATCH["Scratchpad\nSQLite"]:::working --> CONS["consolidate_memory\nuses ctx.sample()"]:::neutral
  EPI["Episodic\nSQLite"]:::episodic --> CONS
  CONS --> SEM["Semantic store\nJSON / Chroma / Azure\nnew FACT-* records"]:::semantic
  CONS --> EPI2["Episodic SQLite\nlog consolidation event"]:::episodic
